# Multilingual Documentation and Runtime Localization

`iztro-rs` requires multilingual documentation and, for user-facing applications, systematic runtime localization. English and Chinese are required at minimum, but they serve different roles.

## Canonical sources

- English documentation is canonical for engineering specifications.
- Chinese documentation is canonical for Zi Wei Dou Shu terminology.
- If behavior and terminology conflict, behavior follows the English engineering spec, while terminology should be checked against the Chinese glossary.

## Required documents

Every major document should have an English and Chinese version where practical:

- README;
- project specification;
- architecture;
- roadmap;
- compatibility policy;
- terminology;
- rule engine design;
- ADRs.

If only one language is updated, the PR should state why. For fast-moving engineering design, it is acceptable to update the English spec first and follow with Chinese terminology/docs cleanup in a later PR.

## File layout

```text
docs/
  en/
    ...
  zh-CN/
    ...
```

Root-level documents may use language suffixes, for example:

```text
README.md
README.zh-CN.md
CONTRIBUTING.md
CONTRIBUTING.zh-CN.md
```

## Runtime localization boundary

The codebase should not assume that Chinese strings are the internal representation. Internal logic should use stable enums, value objects, and keys. Localization should happen at output boundaries.

Implemented runtime direction:

- `crates/iztro-i18n` is the dedicated runtime localization crate;
- Project Fluent resources are used for application-facing text;
- translation resources are bundled at compile time;
- `en-US` and `zh-Hans` are supported first;
- `en-US` is the default GUI/runtime locale;
- Simplified Chinese terminology is preserved as a first-class locale;
- typed helpers cover common domain labels, such as stars, palaces, mutagens, temporal scopes, brightness labels, and shared UI strings.

`iztro-i18n` owns locale parsing, Fluent bundle loading, fallback behavior, and typed label helpers. Application crates such as `iztro-gui`, future TUI, future WASM UI, and future MCP tooling may depend on it.

## Core and facade labels

Core chart facts should remain language-neutral:

```text
StarName::ZiWei
PalaceName::Life
Mutagen::Lu
EarthlyBranch::Yin
```

Localized output should be derived only when rendering or exporting:

```text
i18n.star_name(StarName::ZiWei)
i18n.palace_name(PalaceName::Life)
i18n.mutagen(Mutagen::Lu)
```

Facade/export DTOs may expose additive conventional zh-CN labels for compatibility and readability, especially when matching upstream `iztro` expectations. Those additive labels are not the same thing as runtime localization and should not become the source of truth for GUI text.

## Fallback policy

Runtime localization should be deterministic and safe:

1. If the selected locale has a key, use it.
2. If the selected locale is missing a key, fall back to `en-US`.
3. If `en-US` is also missing the key, return a visible placeholder such as `!missing.key!`.
4. Missing translations must not panic during normal GUI rendering.

The `iztro-i18n` crate implements this for application frontends. It is a Fluent-based localizer that maps typed domain values (`StarName`, `PalaceName`, `Mutagen`, `Scope`, …) and UI strings to localized text at the presentation boundary. It bundles `.ftl` resources at compile time (no runtime filesystem access), defaults to English (`Locale::EnUs`), supports Simplified Chinese (`Locale::ZhHans`), and falls back to English and then a visible `!key!` placeholder for missing keys. The desktop GUI (`iztro-gui`) renders fully in either locale through it; core crates stay language-neutral, and Simplified Chinese values are kept in parity with `core::labels::zh_cn` by tests.

## Review policy

Documentation PRs should identify whether both languages were updated. Runtime localization PRs should identify:

- which locales were changed;
- whether all visible UI strings for the touched surface are covered;
- whether fallback behavior was tested;
- whether any user-facing string remains hardcoded outside translation resources.

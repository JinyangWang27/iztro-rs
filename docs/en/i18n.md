# Multilingual Documentation

`iztro-rs` requires multilingual documentation from the beginning. English and Chinese are required at minimum.

## Canonical sources

- English documentation is canonical for engineering specifications.
- Chinese documentation is canonical for Zi Wei Dou Shu terminology.
- If behavior and terminology conflict, behavior follows the English engineering spec, while terminology should be checked against the Chinese glossary.

## Required documents

Every major document should have an English and Chinese version:

- README;
- project specification;
- architecture;
- roadmap;
- compatibility policy;
- terminology;
- rule engine design;
- ADRs.

## File layout

```text
docs/
  en/
    ...
  zh-CN/
    ...
```

Root-level documents may use language suffixes, for example:

```text
README.md
README.zh-CN.md
CONTRIBUTING.md
CONTRIBUTING.zh-CN.md
```

## Implementation notes

The codebase should not assume that Chinese strings are the internal representation. Internal logic should use stable keys or enums. Localization should happen at output boundaries.

The `iztro-i18n` crate implements this for application frontends. It is a Fluent-based localizer that maps typed domain values (`StarName`, `PalaceName`, `Mutagen`, `Scope`, …) and UI strings to localized text at the presentation boundary. It bundles `.ftl` resources at compile time (no runtime filesystem access), defaults to English (`Locale::EnUs`), supports Simplified Chinese (`Locale::ZhHans`), and falls back to English and then a visible `!key!` placeholder for missing keys. The desktop GUI (`iztro-gui`) renders fully in either locale through it; core crates stay language-neutral, and Simplified Chinese values are kept in parity with `core::labels::zh_cn` by tests.

## Review policy

Documentation PRs should identify whether both languages were updated. If only one language changes, the PR should explain why.

# 工程原则

`iztro-rs` 重视确定性、可测试性、强类型和模块化设计。这些原则用于指导人类贡献者和 coding agents。

## 核心价值

- 先确定性，后表达力。
- 先强类型，后字符串逻辑。
- 先小模块，后大引擎。
- 先显式契约，后隐式约定。
- 解读证据优先。
- 测试是可执行规格。

## 测试驱动开发

对于确定性领域逻辑，强烈鼓励 TDD。

每新增一个排盘规则、特征提取器、规则求值行为或报告渲染行为时：

1. 先添加或更新一个失败测试；
2. 实现最小的确定性改动；
3. 在测试保持通过的前提下重构。

探索性规则知识可以先作为草案数据存在。但一旦某条规则进入维护规则集，就应有测试说明它如何被匹配，以及输出什么结构化判断。

## Rust 风格的 SOLID

本项目采用受 SOLID 启发的 Rust 风格模块化设计。目标不是模仿 Java 或 C# 的类继承，而是用 Rust 惯用法保留有价值的设计约束。

### 单一职责

每个模块应只有一个变化原因。

示例：

- 核心星盘模型不包含解盘文章。
- 特征提取不执行规则匹配。
- 规则求值不渲染最终报告。
- 叙事渲染不重新计算排盘事实。

### 开闭原则

新增方法、派别、四化策略、特征提取器、规则集和叙事风格时，应优先通过 method profile、trait、enum 或数据文件扩展，而不是在大型函数中到处修改。

对于类似插件的可扩展性使用 trait；对于较小且封闭的已知选项使用 enum。

### 可替换性

Rust 没有类继承，但 trait implementation 仍然有契约。一个 trait 的不同实现，应在该 trait 的文档预期内具有行为可替换性。

例如两个排盘策略可以有不同安星规则，但都应该返回有效星盘，包含完整宫位结构、有效星曜落点和一致错误语义。

### 接口隔离

优先使用小 trait，而不是一个巨大的 engine trait。

良好示例：

- `ChartGenerator`；
- `FeatureExtractor`；
- `ReportRenderer`。

避免把排盘、特征提取、规则求值、本地化、规则加载和报告渲染塞进同一个 trait。

### 依赖倒置

高层编排应依赖 trait、method profile 和稳定数据契约，而不是具体实现。

这使得 pipeline 可以用 fixture、mock generator 或替代策略进行测试。

## 测试分类

预期测试类型包括：

- unit tests：纯 helper、索引运算、enum 转换、小型确定性函数；
- golden tests：与选定 `iztro` fixture 的兼容性测试；
- snapshot tests：序列化星盘输出和结构化判断；
- rule tests：从特征输入到命中判断输出；
- integration tests：从星盘到特征、判断、确定性报告的完整链路。

## Coding agent 预期

Coding agent 不应在没有测试或文档的情况下引入大型抽象。搭建脚手架时，应保持层边界，并添加可由测试验证的最小接口。

## 文档预期

影响架构、兼容性、测试政策或公开 API 的工程决策，应通过 ADR 记录。

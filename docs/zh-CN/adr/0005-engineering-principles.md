# ADR 0005：工程原则

## 状态

已接受。

## 背景

`iztro-rs` 将包含确定性排盘逻辑、特征提取、规则匹配、判断聚合和报告渲染。这些领域对边界情况敏感，如果职责混杂，会很难维护。

项目也需要支持 coding-agent 辅助开发。清晰的工程原则有助于让生成代码与架构保持一致。

## 决策

项目采用受 TDD 和 SOLID 启发的 Rust 风格模块化设计原则：

- 确定性领域逻辑应以测试作为可执行规格；
- 对排盘、特征提取、规则求值和确定性渲染，强烈鼓励 TDD；
- SOLID 通过 Rust 惯用法理解，而不是类继承；
- 优先使用小 trait 和小模块，而不是巨型 engine；
- 有意识地使用组合、method profile、trait 和 enum；
- 高层编排应依赖稳定契约，而不是具体实现。

## 影响

- 贡献者应为确定性行为添加测试。
- 新抽象应有明确原因。
- 多派别支持应在适用时使用显式 method profile 和小策略 trait。
- Coding agent 应保持层边界，避免创建大型全能 trait。
- 探索性规则数据可以先不完整测试，但进入正式规则集的规则最终应有规则匹配测试。

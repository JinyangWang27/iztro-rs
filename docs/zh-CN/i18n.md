# 多语言文档

`iztro-rs` 从一开始就需要多语言文档。至少必须支持英文和中文。

## 规范来源

- 英文文档是工程规格的规范来源。
- 中文文档是紫微斗数术语的规范来源。
- 若行为和术语出现冲突，行为以英文工程规格为准，术语应参考中文术语表。

## 必备文档

主要文档都应提供英文和中文版本：

- README；
- 项目规格；
- 架构；
- 路线图；
- 兼容性政策；
- 术语表；
- 规则引擎设计；
- ADR。

## 文件布局

```text
docs/
  en/
    ...
  zh-CN/
    ...
```

根目录文档可使用语言后缀，例如：

```text
README.md
README.zh-CN.md
CONTRIBUTING.md
CONTRIBUTING.zh-CN.md
```

## 实现说明

代码不应假设中文字符串是内部表示。内部逻辑应使用稳定 key 或 enum。多语言本地化应发生在输出边界。

`iztro-i18n` crate 为应用前端实现了这一原则。它是一个基于 Fluent 的本地化器，在展示边界把类型化领域值（`StarName`、`PalaceName`、`Mutagen`、`Scope` 等）与界面字符串映射为本地化文本。它在编译期内联 `.ftl` 资源（运行时不访问文件系统），默认英文（`Locale::EnUs`），支持简体中文（`Locale::ZhHans`），缺失 key 时先回退到英文，再回退到可见的 `!key!` 占位符。桌面 GUI（`iztro-gui`）通过它完整地以任一语言渲染；核心 crate 保持语言无关，简体中文取值由测试保证与 `core::labels::zh_cn` 保持一致。

## Review 政策

文档 PR 应说明是否同步更新了两种语言。如果只更新一种语言，应解释原因。

# 多语言文档与运行时本地化

`iztro-rs` 需要多语言文档；对于面向用户的应用，也需要系统性的运行时本地化。英文和中文至少都需要支持，但二者承担的职责不同。

## 规范来源

- 英文文档是工程规格的规范来源。
- 中文文档是紫微斗数术语的规范来源。
- 若行为和术语出现冲突，行为以英文工程规格为准，术语应参考中文术语表。

## 必备文档

主要文档应尽量提供英文和中文版本：

- README；
- 项目规格；
- 架构；
- 路线图；
- 兼容性政策；
- 术语表；
- 规则引擎设计；
- ADR。

如果只更新一种语言，PR 应说明原因。对于快速变化的工程设计，可以先更新英文规格，之后再用单独 PR 补齐中文术语和文档。

## 文件布局

```text
docs/
  en/
    ...
  zh-CN/
    ...
```

根目录文档可使用语言后缀，例如：

```text
README.md
README.zh-CN.md
CONTRIBUTING.md
CONTRIBUTING.zh-CN.md
```

## 运行时本地化边界

代码不应假设中文字符串是内部表示。内部逻辑应使用稳定的 enum、value object 和 key。多语言本地化应发生在输出边界。

已接受的运行时方向：

- 新增独立 crate：`crates/iztro-i18n`；
- 使用 Project Fluent 资源管理面向应用的文本；
- 使用编译期打包的翻译资源；
- 首批支持 `en-US` 和 `zh-Hans`；
- GUI / 运行时默认语言为 `en-US`；
- 将简体中文术语作为一等 locale 保留；
- 为常用领域标签提供 typed helper，例如星曜、宫位、四化、时间层级、亮度标签和通用 UI 文案。

`iztro-i18n` 应负责 locale 解析、Fluent bundle 加载、fallback 行为和 typed label helper。`iztro-gui`、未来 TUI、未来 WASM UI、未来 MCP 工具都可以依赖它。

## Core 与 facade 标签

核心命盘事实应保持语言中立：

```text
StarName::ZiWei
PalaceName::Life
Mutagen::Lu
EarthlyBranch::Yin
```

本地化输出只应在渲染或导出时生成：

```text
i18n.star_name(StarName::ZiWei)
i18n.palace_name(PalaceName::Life)
i18n.mutagen(Mutagen::Lu)
```

Facade / export DTO 可以保留附加的传统 zh-CN 标签，用于兼容和可读性，尤其是需要对齐上游 `iztro` 习惯时。但这些附加标签不是通用运行时本地化机制，也不应成为 GUI 文本的事实来源。

## Fallback 政策

运行时本地化应确定且安全：

1. 如果选中 locale 有对应 key，使用该 locale 文案。
2. 如果选中 locale 缺少 key，fallback 到 `en-US`。
3. 如果 `en-US` 也缺少 key，返回可见占位符，例如 `!missing.key!`。
4. 普通 GUI 渲染过程中，缺失翻译不应 panic。

## Review 政策

文档 PR 应说明是否同步更新了两种语言。运行时本地化 PR 应说明：

- 更新了哪些 locale；
- 所触及界面的可见 UI 文案是否全部覆盖；
- fallback 行为是否有测试；
- 是否仍有用户可见字符串硬编码在翻译资源之外。
